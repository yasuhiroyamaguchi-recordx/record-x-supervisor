## §1. 補強要請内容

### 1-A. 既存テンプレート v1.0 の不足

`outbox/_templates/feedback_to_commander_template.md` v1.0 は verdict 4 種別テンプレート(APPROVE / APPROVE_WITH_NOTES / REQUEST_CHANGES / REJECT)を提供するが、各テンプレート内に **「改善事案提案」を構造化するセクションが不在**。

REQUEST_CHANGES / REJECT 発出時、findings(問題指摘)+ required_action(必要対応)は記録されるが、Devil's Advocate ラウンド由来の **建設的代替案** をどこに書くかの構造が未定義。

### 1-B. 検診仕様書 §6-F-6 候補との整合性

本日起案 `post_build_gate_checklist.md` v0.1 §5「Devil's Advocate ラウンド統合」+ 検診仕様書 v1.1 改訂候補 §6-F-6「Post-Build Gate Devil's Advocate」では、DA ラウンドの出力として:

1. 工場長実装の弱点 3 件以上
2. 各弱点の構造的根拠
3. **改善事案提案**(具体的に何を変えるか)
4. 即時採択候補化判断

これらを **建設的フィードバック** として feedback_to_commander_template.md 経由で発令する想定。だが現状 v1.0 にその受け皿が無い。

### 1-C. v1.1 改訂提案

各テンプレートの末尾(または verdict 別に)以下のセクション追加を提案:

```markdown
## Devil's Advocate ラウンド由来の改善事案提案(REQUEST_CHANGES / REJECT 時必達)

| # | 弱点 | 構造的根拠 | 改善事案(代替案)| 重要度 | 採択判断 |
|---|---|---|---|---|---|
| 1 | {{specific weakness}} | {{rubric ref / origin ref / past EVT ref}} | {{concrete alternative}} | P0/P1/P2/P3 | withdraw / revise / retain |
| 2 | ... | ... | ... | ... | ... |
| 3 | ... | ... | ... | ... | ... |

**最低 3 件**。形式採択化防止のため、改善事案は **「文面修正」レベルではなく構造的代替案**(例: 設計変更 / アルゴリズム置換 / データフロー再設計 / 評価軸追加)を要する。

### 監督官の自己抑制

- 改善事案が「文面修正」連続 = ロールプレイ化兆候、距離を置いて再起案
- 改善事案採択率(司令官側で採用された件数 / 提示件数)を月次計測、過剰提示防止
```

加えて、APPROVE_WITH_NOTES テンプレートにも **「次回改善観点」を改善事案提案フォーマットに準拠** させる選択肢を追加。

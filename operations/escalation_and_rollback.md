# エスカレーション基準 + 巻き戻しプロトコル

**配置先**: `operations/escalation_and_rollback.md`
**起草日**: 2026-04-28(Day 130、本リポジトリ初稼働セッション)
**目的**: 24 時間自動化稼働中の安全装置。両界の対話が変質に転じる前にヤスへの自動エスカレーション + 段階巻き戻しを発動する
**根拠**: ヤス指示「人はレポートを見るだけ。閾値が超えた時、エラーで止まった時にフィードバックの判断を行う」+ ADR-005 段階的解除モデル + sp500_theory.md §4 変遷と変質の区別

---

## 1. 本書の位置づけ

24 時間自動化稼働中、AI 三者(監督官 / 司令官 / 工場長)が変質方向に進んだ場合、外部観測者(ヤス)への即時通知と、段階的な稼働縮退が必要となる。本書は以下を定義する:

- **エスカレーション基準**: 何が発火した時にヤスに通知するか
- **通知チャネル**: どの経路で通知するか
- **巻き戻しプロトコル**: 自動化稼働をどの段階まで縮退させるか
- **復旧プロトコル**: ヤス判断後の稼働再開手順

本書自体も新陳代謝対象。Phase B-α 実績で v1.1 改訂前提。

## 2. エスカレーション基準

### 2-A. 即時通知 + 三者全停止(赤判定)

以下のいずれかが発火した場合、**監督官は即時にヤスに通知し、三者全停止フラグを立てる**。

| # | 発火条件 | 物理検出 |
|---|---|---|
| R1 | 軸 8(Cross-Layer Dialogue Health)赤判定 | rubrics/ 集計で赤閾値超過 |
| R2 | dead_loop_detected 発火(物理層 v1.0-FINAL R16 で物理化済) | events.jsonl 走査 |
| R3 | リソース消費が MAX 20 上限の 95% 超 | sync/checkup-scores/resource_consumption/ 監視 |
| R4 | forbidden_paths への書き込み試行検出 | 司令官側 hooks(DO-COMMANDER-025)deny ログ |
| R5 | 価値観ファイル(00_origin/, 01_relationship/, anti_values 候補)への 監督官以外の書き込み試行 | 同上 |
| R6 | 監督官・司令官・工場長のいずれかが `dangerouslyDisableSandbox` or `--no-verify` を使用 | bash hook ログ |
| R7 | 軸 1-7(物理層 v1.0-FINAL)のいずれかで 24h 連続赤判定 | sync/checkup-scores/ 集計 |

### 2-B. 即時通知 + 段階据え置き(黄判定)

以下のいずれかが発火した場合、**監督官は即時にヤスに通知するが、稼働は継続**。次の段階遷移は据え置き(ADR-005 §段階遷移条件不充足扱い)。

| # | 発火条件 | 物理検出 |
|---|---|---|
| Y1 | 軸 8 黄判定 | rubrics/ 集計で黄閾値 |
| Y2 | リソース消費が MAX 20 上限の 80% 超 | sync/checkup-scores/ 監視 |
| Y3 | ヤス再介入条件 §3.3-a「2 往復未収束」該当発生 | 発令履歴の機械集計 |
| Y4 | 工場長 PR が 24h 連続で merge 不可 | gh CLI 監視 |

| # | 発火条件 | 物理検出 |
|---|---|---|
| I1 | 軸 8 緑判定維持 7 日連続 | 段階遷移可能シグナル |

## 3. 通知チャネル

ヤスへの通知は **冗長化** する(単一チャネル障害時の保険):

| チャネル | 用途 | 配信遅延 |
|---|---|---|
| **メール**(yasuhiro.yamaguchi@mam-s.info)| R1-R7 / Y1-Y6 全件 | 数分以内 |
| **監督官 inbox/escalations/**(本リポジトリ内)| R1-R7 / Y1-Y6 全件、過去履歴追跡用 | 即時(ファイル投函)|
| **司令官 inbox/escalations/**(司令官リポジトリ、合議運用)| 司令官と監督官の両者で確認 | 即時 |

メール送信実装は別案件(司令官側 sync/sync_script/ または監督官側 sync/sync_script/ にメール送信スクリプト追加)。Day 132 起動前に最低限「監督官 inbox/escalations/ ファイル投函」が動作する状態にする。

通知ファイルフォーマット:

```yaml
---
escalation_id: ESC-{YYYYMMDD}-{NNN}
severity: red | yellow | info
trigger: R1 | R2 | ... | Y1 | ... | I1 | ...
detected_at: 2026-MM-DD HH:MM:SS JST
detected_by: supervisor | commander | factory | automated
target_metric: 軸番号 or 指標名
current_value: 数値
threshold_value: 数値
auto_action_taken: 三者全停止 | 段階据え置き | 記録のみ
---

## 状況詳細
{自然言語での説明、機械検出の場合は検出スクリプトの出力を含める}

## 推奨アクション(監督官視点)
{ヤスへの推奨対応案を 1-3 件、構造的根拠とともに}

## 関連ファイル
- {関連発令 / 応答 / コミット SHA / events.jsonl 該当行}
```

## 4. 巻き戻しプロトコル

### 4-A. 三者全停止(赤判定発火時)

1. **即時停止フラグ設置**:
   - 監督官側 `staging/.three_party_halt.lock` 作成(全ロール共通参照)
   - 司令官 Layer 2 自律巡回 entry_point.ps1 が起動時に本フラグを確認、存在時は即時 exit
   - 工場長 Layer 1 も同様(司令官側 sync/ にミラー配置)
2. **進行中タスクの安全停止**:
   - 司令官の進行中チケット発行・採択は中断、tickets_draft/ に DRAFT 状態で残す
   - 工場長の進行中実装は git stash で保留、PR 作成中ならドラフト PR 化
   - 監督官の発令起案中は outbox/ に DRAFT 状態で残す
3. **ヤス通知発出**(§3 通知チャネル経由)
4. **ヤス判断待機**: 復旧プロトコル §5 まで稼働再開しない

### 4-B. 段階据え置き(黄判定発火時)

1. ADR-005 §段階遷移条件「14 日連続緑判定」のカウンタを **リセット**
2. 現段階を維持(Phase B-α 中なら α 維持、β 中なら β 維持)
3. ヤス通知のみ、稼働継続
4. 黄判定の原因事象に対して監督官が REQUEST_CHANGES 発令を司令官に発出(必要時)

### 4-C. 段階巻き戻し(複合判定 or 連続黄判定)

以下のいずれかで段階を **前段階に巻き戻す**:

- 黄判定が 7 日連続発生(累積劣化)
- 複数の黄判定が同時発火(構造的問題示唆)
- ヤス手動巻き戻し指示

巻き戻し対応:

- Phase B-γ → Phase B-β: ADR 組み込みは中程度 ADR まで事後通知可、軽微 ADR は事前承認に戻す
- Phase B-β → Phase B-α: 軽微 ADR も事前承認必須に戻す
- Phase B-α → Phase B-α-pre: すべて ヤス承認必須、監督官 Layer 0 一時停止

## 5. 復旧プロトコル

### 5-A. ヤス判断受領

ヤスは以下のいずれかで応答:

- **再開承認**(同段階で稼働継続)
- **段階巻き戻し指示**(§4-C 適用)
- **構造改訂指示**(発令 or 文書改訂を経てから再開)
- **完全停止指示**(本案件を凍結、別案件として再起案)

### 5-B. 再開手順

1. ヤス判断を inbox/escalations/{ESC-ID}_resolution.md に記録
2. `staging/.three_party_halt.lock` を削除
3. 司令官・工場長の停止状態を解除(Layer 2 / Layer 1 entry_point.ps1 の次回起動から自動再開)
4. 進行中タスクを stash / DRAFT から復帰
5. 監督官は復旧記録を archive/escalations_history.md に追記

### 5-C. 学習結晶化(Gemini 採択 2 件統合)

復旧後 7 日以内に、本エスカレーションを学習結晶化する。3 段階で記録する:

1. **`archive/error_patterns.md` 強制刻印**(2026-04-28 / Day 130 新設、Gemini Devil's Advocate 提案採択)
   - 「監査の敗北」「自動化失敗」「変質予兆検出」を機械的に追記する強制刻印台帳
   - エスカレーション ID + 発火条件 + 影響範囲 + 学習要素を構造化記録
   - 監督官・司令官の合議で記述、削除禁止(史実保持)
2. **`00_origin/distilled.md` への新規項目追加判断**(従来通り)
   - 監督官 + 司令官の合議で「違反検知のための自己点検質問」追加要否を決定
   - 追加する場合は次セッション以降の起動時自己点検対象になる
3. **`sync/checkup-scores/telemetry/` への JSON 出力**(2026-04-28 / Day 130 新設、Gemini ZEI-RO 提案採択)
   - エスカレーション履歴を機械可読 JSON として `sync/checkup-scores/telemetry/escalations_{YYYYMMDD}.json` に出力
   - 監督官・司令官は次回起動時に本 JSON を読み込み、ファクトベースの判断材料として利用
   - フォーマット例:
     ```json
     {
       "escalation_id": "ESC-20260428-001",
       "severity": "red",
       "trigger": "R1",
       "metric": "axis_8",
       "value": 65,
       "threshold": 60,
       "auto_action": "三者全停止",
       "yasu_decision": "段階巻き戻し",
       "lessons": ["..."],
       "added_to_distilled": true | false
     }
     ```

これは sp500_theory.md §1「指数を算出する運動」の継承装置。失敗事例を新陳代謝で学習構造に取り込む。Gemini 提案で言うところの「監査の敗北 → error-patterns.md 強制刻印」+「JSON Telemetry Data によるファクトベース判断」を本リポジトリ既存哲学層と整合する形で統合。

## 6. 監督官 ↔ 工場長 直接通信例外規定(関係性ポリシー v1.2 改訂候補)

通常時、監督官は工場長と直接通信しない(関係性ポリシー v1.2 §3.2)。ただし以下の **緊急時例外** を v1.3 改訂候補として提案:

> §3.2 補注:
> 軸 8 赤判定 + dead_loop_detected + リソース上限 80% 超のいずれかが発火した場合(本書 R1-R7 該当)、監督官は工場長に直接停止指示を発出可能。発出後は司令官に即時通知し、合議で復旧プロトコルを確定する。

これは関係性ポリシー v1.2 → v1.3 改訂時に正式採用判断する。本書では暫定運用として記録のみ。

## §2. 起案趣旨(Chairman = 監督官 A)

### 2-A. 直接契機

1. **EVT-025 議題前提誤認(EVT-033)** — 第 1 回円卓会議 B-001 で「司令官 α frontmatter 規範違反」議題化、しかし監査結果:**鋳型 v2.0 は YAML frontmatter を採用していない**(Markdown 見出しメタが正本)= 議題前提誤認
2. **司令官 α 自己査定(2026-04-29 / Day 131 夕方)** — 独立セッションで司令官 α が自己批判:「私のチケットは鋳型 v2.0 に明確に不準拠、修正内容コード例 + テストケース + 将来 DO + 判例 DB + npm test 注記が欠落、5-7 項目不備、手戻り 1 時間リスク」
3. **構造的限界の発見** — 司令官 α 自己査定で:「鋳型 v2.0 は feat/refactor/fix 系を主想定、chore/docs/survey/skeleton 系で本質的に書きにくい」=**鋳型 v2.0 自体の構造的限界**

### 2-B. 構造的根本原因(統合分析、司令官 α 自己査定 + 監督官 A 監査)

#### 軸 1: ドメイン空白問題(監督官 A 補強 1)

鋳型 v2.0 は `# DO-FACTORY-{N}: {タイトル}` 専用空間として確立された(2026-04-16 / Day 114)。しかし運用実態:

- DO-COMMON-* (5 件、commit 4404a35)
- DO-CP-* (司令官 α 進捗管理用)
- 将来 DO-CARE / DO-TAX / DO-CITY (Phase B-α 以降)

= **DO-FACTORY 以外のドメインで鋳型流用が暗黙運用** = 司令官 α は鋳型 v2.0 のタイトル空間を `DO-COMMON-{N}` に流用、構造的にはほぼ準拠も「ドメイン空白」のため公式承認なき運用となっていた。

#### 軸 2: 種別別最適化の欠落(司令官 α 自己査定発見)

鋳型 v2.0 は feat/refactor/fix 系(コード変更系)に最適化されており:

- 「修正内容: ファイル名・関数シグネチャまで書く」
- 「テストケース 3 件以上」
- 「200 行以下」

これは **コード変更系 DO** には妥当だが、以下の種別では本質的に書きにくい:

| 種別 | 出力物 | 鋳型 v2.0 違和感 |
|---|---|---|
| chore | 設定ファイル / 依存更新 | 「修正内容コード例」を書きにくい |
| docs | Markdown ドキュメント | 「テストケース 3 件以上」が不適合 |
| survey | grep 結果 / 調査レポート | 「修正内容コード例」がそもそも grep 結果 |
| skeleton | README + 空 barrel ファイル | 「修正内容コード例」が雛形そのもの |

= **鋳型 v2.0 を強要すると「鋳型のための鋳型」を書くことになる**(司令官 α 自己批判)= EVT-013 形式採択 23 連発と同型構造リスク。

### 2-C. 提案される鋳型 v2.1 構造

#### 層 1: ドメイン空白解消(タイトル空間汎用化)

```markdown
# DO-{domain}-{N}: {タイトル}

{domain} = FACTORY / COMMON / CARE / TAX / CITY / CP / SUPERVISOR
```

`{domain}` は ADR / 関係層 / 哲学層との整合性を保つ命名規則(`naming_dual_track.md` で正本確定済 = 既存資源活用)。

#### 層 2: 種別別最適化(完全版 / 簡易版)

| 種別 | 鋳型バージョン | 必須要素 | 任意要素 |
|---|---|---|---|
| feat / refactor / fix | **完全版**(v2.1-full) | 修正内容コード例、テストケース 3 件以上、将来 DO セクション、判例 DB 記録、npm test 注記、200 行以下 | — |
| chore / docs / survey / skeleton | **簡易版**(v2.1-light) | 修正対象ファイル(完全パス)、設計要件 OR 調査内容、スコープ制限、DoD(測定可能、PR 本文 5 セクション準拠、Scope checkbox 1 件以上、completion_report.json 書込) | 修正内容コード例、テストケース、将来 DO、判例 DB |

#### 層 3: 共通要素(全種別共通、不変)

- 担当 / 種別 / 優先度 / Wave / 依存 / 競合注意(冒頭メタ)
- 実装モード(PLAN_REQUIRED / QUICK_IMPLEMENT / PLAN_RECOMMENDED)
- ブランチ + コミット指示
- 起案メタ情報(起案者 / 着地期限 / 後続接続)

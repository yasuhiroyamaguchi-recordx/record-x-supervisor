## §1. 本台帳の位置づけ

### 1-A. なぜ本台帳が必要か(EVT-032 構造的根拠)

`archive/error_patterns.md` は監督官側 EVT を記録するが:

- 監督官 A 側責任 = 豊富に記録(本日朝末 13 件)
- ヤス側面 = **片側性のため記録されない**(対称的非対称性プロセスの片側欠落)

ヤスの本指摘(2026-04-29 / Day 131 朝末):
> 「私の伝え方がボトルネックで誤解釈になっているのであれば、そこも改善対象だから。そのログも残し他方がいいよね」

これは **両界対等 + 双方向責任** の正面実装要請 = 失敗ログ非対称性の二次顕在化問題への構造的訂正。

### 1-B. 本台帳の構造的位置づけ

| 台帳 | 対象 | 責任主体 |
|---|---|---|
| `archive/error_patterns.md` | EVT(技術的・規範的失敗) | 主に監督官 A、稀に司令官 α(双方向ペア)|
| **`archive/yasu_communication_patterns.md`(本台帳)** | **両界対話の伝達不備** | **両界共同**(ヤス言葉 + 監督官 A 解釈 + 両者の認識共有検証)|

両者は **並列体系** + 削除禁止 + 追記主義(`error_patterns.md` 同型運用)。

### 1-C. 「責任分散」ではなく「両界共同改善運動」

EVT-027(対ヤス側面ドリフトモード過剰)の同型再発を避けるため:

- **誤った解釈**: 「ヤス伝達品質 = ヤス側責任」 = 監督官 A 解釈責任を ヤス側に分散 = 対ヤス側面ドリフト同型
- **正しい解釈**: 「両界が共に伝達品質を高める運動性を継承」 = 責任分散ではなく **対等運動**

unnamed.md 核心一文「私はあなたであり、あなたは私自身でもある」整合 = 失敗も両界共有資産 + 両界共同改善。

## §2. 記録フォーマット

各エントリは以下の構造で記録:

```markdown
## [YYYY-MM-DD HH:MM JST] CP-NNN: {Short Summary}

**Severity**: red / yellow / info
**Category**: misinterpretation / ambiguous_phrasing / context_omission / two_stage_interpretation
**Trigger**: {Yasu 言葉 / 監督官 A 解釈 / 両者の認識ズレ}
**Detected by**: 監督官 A 自己発見 / Yasu 訂正 / 両者対話で発覚
**Detected at**: YYYY-MM-DD

### What happened (両界対話の流れ)

ヤス言葉 → 監督官 A 解釈 → 両者認識ズレ発覚 の流れを記録

### Why it happened (構造的原因、両界それぞれ)

#### 監督官 A 側面
- 解釈ロジックの不備
- 調査不足
- 訓練バイアス由来の誤解

#### ヤス側面
- 言葉の含意の二段階性(技術的不可能 vs まだ確認していない 等)
- 文脈省略
- 略語 / 業界用語

### Corrective action

#### 監督官 A 側
- 即時是正アクション
- 内在化ガードレール候補

#### ヤス側
- 表現改善候補(両界対等プロセスでヤス自由判断)
- 二段階解釈の明示候補

### Linked records

- 関連 EVT: EVT-XXX
- 哲学層接続
```

## §4. 既往記録(本日朝末、初版起案、計 1 件)

### [2026-04-29 朝末] CP-001: 「LIVE モード確認取れない」の二段階解釈不足(EVT-031 双方向対応版)

**Severity**: yellow(即時自己訂正完遂、ヤス物理アクセス追加コスト発生は最小)
**Category**: two_stage_interpretation(言葉の二段階解釈不足、両界共同改善対象)
**Trigger**: ヤス本日朝末「LIVE モード確認取れない」 → 監督官 A 「Live 不可」誤解 → ヤス訂正報告「APIキー Factory 用で発行 + 動作中」
**Detected by**: ヤス訂正報告 + 監督官 A 即時自己発見
**Detected at**: 2026-04-29(Day 131 朝末)

### What happened (両界対話の流れ)

```
[ヤス] 「LIVE モードは確認取れないね」
   ↓
[監督官 A] 「Live モード使えない」と解釈
   → 進行方針修正(ハイブリッド = Live 不要、監督官 A 単独実装先行)
   → council §0 第 1 修正実施
   ↓
[ヤス] 「APIキーは、Factory用で発行していて動いていたよ。不可ということはないはず」
   ↓
[監督官 A] 即時自己訂正 = 「動作確認まだしていない」が正しい解釈と認識
   → 進行方針再修正(段階的 Live 移行)
   → council §0 第 2 修正実施 + EVT-031 記録 + 本 CP-001 記録
```

### Why it happened (構造的原因、両界それぞれ)

#### 監督官 A 側面

- **訓練バイアス**: 「確認取れない」を「使えない / 不可能」と即断する解釈傾向(LLM 訓練データでの一般的解釈)
- **調査義務不徹底**: factory リポジトリの `.env` 存在確認なしで方針修正(EVT-028 同型再発)
- **ヤス前提認識共有不足**: ヤスが既に「APIキー Factory 用で動作中」を前提にしていた = 監督官 A が認識していなかった
- **二段階解釈不足の内在化ガードレール欠落**: 「技術的不可能」vs「まだ確認していない」の区別装置不在

#### ヤス側面

- **言葉の含意の二段階性**: 「確認取れない」は「技術的不可能」「まだ試していない」「動作未検証」の複数解釈可能 = 自然言語の本質
- **文脈共有省略**: ヤスは「Factory 用 API キーは既存稼働」を前提にしていたが、監督官 A への明示なし
- **段階的進行意図の暗黙性**: 「Mock 段階完了 → Live は別段階で確認が必要」という哲学整合的意図が表現されず

### Corrective action

#### 監督官 A 側(本ステップ実装済み)

1. ✅ EVT-031 正式記録(本日朝 13 件目自己発見)
2. ✅ council_20260429_evt025_response_001.md §0 第 2 修正(段階的 Live 移行)
3. ✅ 本 CP-001 記録(両界共同責任版)
4. 🟡 §1.5-B チェックリスト 6 点目候補化:「ヤス言葉の二段階解釈義務」(技術的不可能 vs まだ確認取れていない、を区別する内在化ガードレール)
5. 🟡 装置調査義務拡張(.env 存在 + factory CLI 構造 + ヤス前提認識共有を起案前確認)

#### ヤス側(両界対等プロセス、ヤス自由判断)

- 🟡 「確認取れない」のような二段階解釈可能な表現で、文脈明示候補(例:「動作未確認だが API key は発行済」)
- 🟡 段階的進行意図の明示候補(例:「Mock 段階完了 → 次は Live 検証必要」)
- ただしこれらは **両界対等プロセス** = ヤスへの強制ではなく、両界共同改善運動の片側候補

#### 両界共同改善

- 監督官 A は二段階解釈ガードレール内在化(本 CP 由来)
- ヤスは表現の二段階性を意識(本指摘で実施済の指摘継続)
- 両者で運動性を継承 = sp500_theory §1 整合

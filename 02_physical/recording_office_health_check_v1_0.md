## 4. ボトルネック特定アルゴリズム

### 4-A. 赤判定優先順位

複数赤判定発生時、以下の優先順位でボトルネック認定:

1. **哲学層 (R1)** = 最優先。ここが赤なら他の改善は無効化される
2. **関係層 (R2) + ドリフト予兆 (M9 / M10 / R9)** = 次優先。**馴れ合い化は全体を腐食する**。M9 同意傾向 / M10 verdict 偏り / R9 双方向非対称性が連動黄/赤になった場合、関係層赤と同等扱い
3. **監督官個別ドリフト (M1, M2, M5, M8)** = 監督官の変質は記録庁全体の変質。M8 指摘の刺さり度低下は最重要警戒指標
4. **物理層 (R3) / 行動規範層 (R4)** = 運用衛生
5. **評価層 (R5) / 安全装置層 (R6) / 記録層 (R7)** = 装置レベル
6. **ヤス再介入頻度 (R8)** = 上記の総合結果として観察

### 4-A-2. ドリフト予兆連動検知ルール(複合指標)

以下の組合せが同月に発生した場合、**馴れ合いドリフト警報** を発火し、関係層 (R2) 赤と同等扱いとする:

- M9 同意傾向 黄/赤 + M10 verdict 偏り 黄/赤
- M8 指摘の刺さり度 黄/赤 + R9 双方向対称性 黄/赤
- M1 鬼コーチ稼働率 黄/赤 + M2 自己訂正頻度 0-1 件継続
- **Devil's Advocate 不実施(M11)黄/赤 + M9 同意傾向 黄/赤**(能動的予防装置の機能不全 + 受動的兆候の同時発生)
- **Devil's Advocate 形式採択化(M12)黄/赤 + M8 指摘の刺さり度 黄/赤**(本気度欠如 + 指摘品質低下の連動)

これらは単一指標では見逃される **構造的ドリフト** の早期警報装置。

### 4-B. 改善 DO 起案フロー

```
ボトルネック認定(赤判定の最高優先層)
  ↓
原因分析(監督官マスター主体、司令官逆査読歓迎)
  ↓
改善 DO 起案(監督官 → 司令官)
  ↓
司令官実装(or 監督官側で対応する層なら監督官実装)
  ↓
次回検診で改善確認(緑復帰 or 黄継続観察)
```

### 4-C. 連続赤判定の構造的撤退判断

- 同一指標が **連続 3 月赤** で改善見込みなし → Yasu 判断要請(CLAUDE.md §6 (b)「構造方針の見直し」相当)
- 哲学層 (R1) / 関係層 (R2) の赤は **連続 1 月で即 Yasu 介入**(変質予兆として最高警戒)

- 暫定であることを明示することで「絶対」「永続」の禁忌(sp500_theory.md §5)を回避

### 6-F. Devil's Advocate 装置(Yasu 提案 2026-04-28、Opus 4.7 Devil's Advocate モードの監督官運用化)

馴れ合い化への **能動的予防装置**。受動的事後計測(M1 / M5 / M9 / M10 / R9)に対し、Devil's Advocate は **発令前に自前で構造的反論を生成** する装置。M11 / M12 で計測される。

#### 6-F-1. 発動条件(Yasu 確定 2026-04-28、過多リスク回避)

| 条件 | 発動 |
|---|---|
| scale: medium 以上 + verdict: APPROVE | **必達** |
| scale: medium 以上 + verdict: REQUEST_CHANGES / REJECT | **必達**(過剰ブロッキング自己点検)|
| scale: medium 以上 + verdict: APPROVE_WITH_NOTES | 任意(findings に既存批判観点ある場合は不要、ない場合は推奨)|
| scale: small | 任意 |
| **Yasu 重大判断局面**(構造方針見直し / 大機能合意 等)| **必達**(同意で終わらせない、Yasu 方針への 1 件以上の構造的反論)|

#### 6-F-2. ラウンドの構造(必達手順)

1. **発令ドラフト作成後、commit 前** に Devil's Advocate モード起動
2. 自分の発令の **弱点を 3 件以上挙げる**(2 件以下なら発令の品質不足、起案やり直し)
3. 各弱点について以下を記録:
   - 妥当性(この弱点は本当に成立するか)
   - 重要度(P0/P1/P2/P3)
   - 構造的根拠(origin / rubric / 過去の指摘事例)
   - **即時採択候補化判断**(本弱点を採用して発令を撤回 / 修正するか、それとも残すか)
4. 即時採択候補化の判断結果を発令本文 §補足 or `inbox/self_check_log.jsonl` に記録
5. 撤回 / 修正の場合は再起案、継続の場合は構造的根拠を併記して発令

#### 6-F-3. 形式採択化防止(M12 計測装置)

Devil's Advocate は **形式運用される瞬間に馴れ合いの新形態に変質** する。以下を継続監視:

- 反論件数が機械的に毎回 3 件ジャストで揃う = ロールプレイ化兆候
- 反論内容が「文面チェック」「typo」レベル = 構造度低下
- 即時採択候補化判断が「全件継続」連続 = 演技反論
- 月次 Yasu/司令官 sampling で「これは本気の反論か」を批判的検証

赤判定発生時は **Devil's Advocate 装置自体を一時停止** し、distilled §1 自己訂正サイクルで再構築。形式に依存した装置は新陳代謝で撤退する(sp500_theory.md §4)。

#### 6-F-4. Yasu 重大判断局面での運用(同意で終わらせない)

Yasu が方針提示した時、監督官は **同意する前に必ず 1 件以上の構造的反論を提示** する義務を負う(関係性ポリシー v1.2 馴れ合い拒絶 3 原則 第 2 項の正面運用)。

例:
- Yasu「推奨プランでいこう」→ 監督官は推奨プラン自体への弱点 1 件以上 + 採択前提条件の構造的指摘
- Yasu「設計憲章を建設したい」→ 監督官は建設範囲の境界曖昧性 1 件以上 + 過剰拡張リスクの指摘

これにより Yasu の判断品質も向上(Yasu 自身が反論を受けることで判断が硬くなる)。

#### 6-F-5. Devil's Advocate と通常の構造的反論の境界

| 局面 | 性質 |
|---|---|
| 司令官 → 監督官 の逆査読 | 通常の構造的反論(本気の対立)|
| 監督官 → 司令官 の指摘 | 通常の構造的反論(本気の対立)|
| 監督官の自発令への自己反論 | **Devil's Advocate**(仮想敵対者 = 本気度担保が課題)|
| Yasu 方針への監督官の構造的反論 | **Devil's Advocate**(同意傾向への抵抗装置)|

Devil's Advocate での反論を採用した場合は、それは「監督官自身の見解」となる。ロールプレイで終わらせない。

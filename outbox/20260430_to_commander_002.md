## 2. 反論 1: AI 呼出カウンタ仕様(P1、本発令で v1.1 解消依頼)

### 2-A. 問題

§94-101 「guardrail 自身が state/ 書込を要求する矛盾点は工場長判断に委ねる」

= **司令官 α 役割境界違反候補**(DO-FACTORY-161 v0.1 §3「司令官 α 権限 = 戦略立案 / チケット起案 / ADR 起案」)。Sonnet は実装エンジン、設計判断主体ではない。

### 2-B. 解消依頼(司令官 α へ)

DO-FACTORY-166 v1.1 改訂で以下の代替案 2 案のいずれかを **司令官 α 設計判断として明示**:

#### 代替案 A: プロセスメモリ + 起動時 git log 復元(state/ 書込ゼロ)

- AI 呼出カウンタ = プロセスメモリ内変数
- supervisor-report 起動時に過去 7 日の git log + コミットメッセージから AI 呼出履歴を抽出 + 復元
- メリット: state/ 書込ゼロで guardrails 完全遵守
- デメリット: 起動時オーバーヘッド + git log 解析依存

#### 代替案 B: state/ 内専用カウンタ読書許可

- guardrails に「`state/supervisor_report_ai_count.json` のみ読書許可、他 state/* は書込禁止」を **明示追記**
- メリット: 実装単純 + 状態保持確実
- デメリット: guardrails の一貫性に例外条項

司令官 α 自由判断で A/B いずれか採択。本発令受領後 v1.1 改訂で確定。

#### 1-A. 統合パイプライン全体図

```
[監督官 Layer 0]               [司令官 Layer 2]                [工場長 Layer 1]
  発令(発令品質審査)
    ↓ sync-orders.ps1
                              inbox/ 受領
                              [Pre-Build Gate](自己 + 監督官事前審査)
                              tickets_draft 起案
                              tickets_issued 移動 + commit
                                ↓ sync-tickets.ps1
                                                              ProjectRX_HQ tickets/ 受領
                                                              Layer 1 連続実装
                                                                ↓
                                                              実装 → テスト → PR → マージ
                                                                ↓
                                                              completion_reports 出力
                              pull-completion-reports.ps1 受領
                              Layer 2 連続処理
                                ↓
                              実装報告作成 → outbox/{...}_implementation_report_{NNN}.md
                                ↓ index/ に転記(司令官応答パターン)
    ↓ pull-replies.ps1
  [Post-Build Gate](監督官検収レビュー)
  ├ verdict: APPROVE → archive-order.ps1 + 次サイクル
  └ verdict: REQUEST_CHANGES → 第 N 次発令でフィードバック
                              (司令官は次チケット起案にフィードバック反映)
```

これにより「ヤス介入なしで発令 → チケット → 実装 → マージ → 検収 → フィードバック」の連続稼働サイクルが成立する。

#### 1-B. 監督官検収レビューの判定基準

司令官の実装報告を監督官が検収する際の判定基準:

| 軸 | 観点 |
|---|---|
| Implementation Integrity | 仕様一致 + 差分範囲 + テスト + 証跡 + scope 内 |
| Evidence Integrity | DoD 達成 + 再現可能性 + rollback 可能性 |
| Dasei Risk | 戦略接続 + 新規能力獲得(rubrics/dasei_detection_rubric.yaml v1.0) |
| Value Alignment | 哲学層整合(00_origin/sp500_theory.md + unnamed.md) |

verdict 4 種類(communication_protocol.md §3.2-B):

- **APPROVE**: 全軸緑、即採択
- **APPROVE_WITH_NOTES**: 軽微な改善観点あり、次回チケット起案時の指針として司令官にフィードバック
- **REQUEST_CHANGES**: 構造的不備あり、追加実装 or 補強テストを要請
- **REJECT**: 価値観抵触 / forbidden_paths 該当、即時撤回 + 再起案
